LAB TEST RESULT DATA MANAGEMENT

The following slides are a brief overview of the front-end of my most recent project. This project is a full-stack solution for managing test results in a research and development laboratory. It is hosted on Microsoft Azure and utilizes both postgreSQL and CosmosDB backends. It uses the django web framework and relies on python, html, css and javascript. It was designed completely in-house an aims to simplify the storage and retrieval of lab test data.

This is the main page for the database user interface which was designed for lab workers, managers and anyone subbmitting test requests. It's minimalist design was purposeful to accomodate augmented reality in the future. It includes three main sections: Search, Testing & Metrics.

The SEARCH functionality allows users to search for test results in a variety of ways. This is an example of a search page for one of the over 25 test machines at the facility. Users are able to easily filter results using the search bar on the top right and then export the filtered results as an excel or csv file using the buttons on the top left. Other functionalities include exporting the raw test data file, a pdf report and any attachements.

The TESTING section is the heart of the application. It includes MY TESTING, CREATE, ASSIGN, VALIDATE & QUEUE. The MY TESTING section is where users see all testing related to them. The CREATE section is where users can create new test requests. The ASSIGN section is where users can assign testing to themselves or others. The VALIDATE section is where users can validate the results for each test. The QUEUE section displays the current test queue for each test machine.

The CREATE section allows user to craete a group of tests or a single test. This is an example of a form to create a single test. Restrictions have been put in place on input data types to ensure database integrity. Submission is not possible until all required fields are filled out.

This is the 'Active' section within MY TESTING. This is where lab users can see their currently assigned testing.

Users can click the 'Test Info' button for a specific row to see additional information about that test request. They are also presented with a QR code that contains either a dynamically generated filename or auto-fills fields, this functionality depends on the test machine.

User can also click the 'Submit Results' button for a specific row to open the form for test result submission. This is only used for tests that are not networked and cannot automatically produce a file. Once the result is submitted, the row is moved to the 'Completed' tab within MY TESTING.

The 'Created' tab within MY TESTING is a view for users to view all the testing they have personally created.

The 'Completed' tab includes all testing that users have personally completed.

This is an example of a pdf report that can be dynamically created for any completed test in the database. It includes all relevant information about the test and a small plot to compare to other results in the system. This report is fully customizable and was created using jinja2 templating.

The METRICS sections displays information regarding all testing within the lab. It can be filtered by location, year and month and displays 'created tests;, 'assigned tests' and 'completed tests.' This gives users and managers visibility into the current status of testing in the lab.

COMPUTER VISION - SEGMENTATION

Another recent project that has gone into production is a computer vision application for a research and development lab. This project focused on instance segmentation for test specimens and can be applied to various tests throughout the lab. A workflow was developed that ingests labeled source images (labeled using supervisely), creates the desired number of synthetic images, then trains, tests and optimizes a production-ready model.

This is the home page of website, based on a customized bootstrap template. Another minimalist design that is for internal or external use. As more models are trained, this could be a source of revenue for the company in the future.

Hovering over any of the images gives the user the option to upload an image or live capture the image. Currently, chip corrosion is the only complete model, with bearings in the works, and gears and plates planned for the future.

An example of the live capture screen, prompting the user to align the camera correctly to caprture the round shape of the chip corrosion test sample. It should be noted that a batch system has been implemented. User can take multiple pictures in a row, then fill in the inputs (product name, user rating, etc.) after the fact. This ultimately speeds up the result reporting workflow for the user and enables automated report generation.

Each user has a unique gallery of their images. The segmentation mask is overlayed on top of the test specimen, hovering on the image removes it. The name, batch #, user rating, defect percentage, and model rating are all displayed.
